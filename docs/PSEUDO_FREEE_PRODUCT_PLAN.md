# 疑似freee プロダクト実装計画

> 📌 **発展方針の正本は [`EVOLUTION_PLAN.md`](EVOLUTION_PLAN.md) です。** 本書は pseudo_freee の機能詳細です。全体の進め方・採用スタックは正本を優先してください。

## 目的

`pseudo_freee` は、単なる受信データの表示板ではなく、在庫管理から連携された仕入・売上と、手入力の経費をまとめて扱う会計ダッシュボードとして育てる。

本物のfreee APIへ進む前に、以下の流れをローカルで説明・デモできる状態を目指す。

```text
inventory_dashboard
  仕入・売上登録
  freee_sync_queueへ送信用JSON保存
  疑似freeeへ送信

pseudo_freee
  会計取引として受信・保存
  手入力経費も登録
  KPI・月次推移・取引一覧・詳細で確認
```

## 現状

実装済み:

- `POST /api/deals` で在庫管理側の会計データを受信
- SQLite `pseudo_freee.db` へ保存
- 取引一覧画面
- 取引詳細画面
- 受信JSON表示
- `queue_id + source_type + source_id` による二重送信防止
- 手入力経費
- KPI、未入金・未払、月次推移
- 取引検索・フィルタ
- `inventory_dashboard` から `pseudo_freee` への実送信
- 支払先・勘定科目・税区分の候補入力

不足しているもの:

- 請求書・レシート画像アップロード
- スマホカメラ撮影入力
- AI/OCRによる経費入力候補作成
- 取引詳細での添付画像、AI抽出結果、ユーザー修正後データの確認

## 完成イメージ

在庫管理側で仕入・売上を登録すると、`freee_sync_queue` に送信用JSONが保存される。ユーザーが送信前JSONを確認し、「疑似freeeへ送信」を押すと、`pseudo_freee` の `POST /api/deals` に送信され、会計取引として保存される。

疑似freee側では、在庫連携された仕入・売上に加えて、手入力の経費を登録できる。ダッシュボードでは今月売上、今月仕入、手入力経費、粗利、未入金予定額、未払予定額、取引数を確認できる。取引一覧では検索・フィルタを使って、収入・支出・取引先・日付範囲で確認できる。

## 実装方針

### pseudo_freee

- 在庫連携取引と手入力取引を同じ会計取引テーブルで扱う。
- `source_app` は `inventory_dashboard` または `manual` とする。
- `source_type` は `purchase`、`sale`、`manual_expense` を扱う。
- 手入力経費では `queue_id` と `source_id` を空にできる。
- 手入力経費の項目は、発生日、支払予定日、支払先、勘定科目、税区分、金額、摘要、メモとする。
- 支払先と勘定科目は検索付き候補入力にする。
- 税区分はドロップダウンにする。
- 候補にない支払先・勘定科目は新規入力でき、登録後に候補へ追加する。
- 受信JSONまたは手入力スナップショットを `payload_json` に保存する。

### 証憑AI入力

- 請求書・レシート画像アップロードを追加する。
- スマホではカメラ撮影または画像選択ができるようにする。
- AI/OCRは直接登録せず、発生日、支払先、金額、税区分、摘要、勘定科目候補をフォームへ仮入力する。
- ユーザーが確認・修正してから経費登録する。
- 詳細仕様は `docs/EXPENSE_CAPTURE_FEATURE_SPEC.md` に記載する。

### inventory_dashboard

- `PSEUDO_FREEE_API_URL` を環境変数で読み、未指定時は `http://127.0.0.1:8010` を使う。
- `freee_sync_queue` の `payload_json` を `POST /api/deals` へ送信する。
- 成功時は `status = sent`、`external_accounting_id = pseudo-freee-{id}` に更新する。
- 失敗時は `status = failed`、`sync_error_message` に理由を保存する。
- 送信済みキューは再送しない。

## 実装順序

1. この計画書を作成する。
2. `pseudo_freee` のSQLiteスキーマを手入力取引に対応させる。
3. 経費入力フォームと保存処理を追加する。
4. KPI、未入金・未払、月次推移を追加する。
5. 取引一覧に検索・フィルタを追加する。
6. `inventory_dashboard` に疑似freee送信APIクライアントと送信ボタンを追加する。
7. 送信成功、失敗、重複、再送の表示を整える。
8. READMEと要件定義書に、現在の実装範囲と今後の構想を反映する。
9. 経費入力の支払先・勘定科目・税区分を候補マスタ化する。
10. 請求書・レシート画像アップロードとスマホカメラ入力を追加する。
11. AI/OCRによる経費入力候補作成を追加する。
12. 取引詳細で添付画像、AI抽出結果、ユーザー修正後データを確認できるようにする。

## テスト観点

- 在庫連携の `POST /api/deals` で仕入・売上が保存される。
- 同じ `queue_id + source_type + source_id` を再送しても二重登録されない。
- 手入力経費が保存され、一覧・詳細・KPI・月次推移に反映される。
- 受信JSONまたは手入力スナップショットが詳細画面で確認できる。
- `inventory_dashboard` のpendingキューを疑似freeeへ送信できる。
- 成功時に `freee_sync_queue.status` が `sent` になる。
- 失敗時に `failed` とエラーメッセージが残る。
- 送信済みキューを誤って再送しない。
- 支払先・勘定科目・税区分の候補が表示される。
- 新規入力した支払先・勘定科目が次回候補に追加される。
- 将来の証憑AI入力では、AI推定結果を直接登録せずフォーム反映に留める。

## 前提

- 本物freee APIの完全再現ではなく、ポートフォリオ用の会計連携デモアプリとして作る。
- 経費入力はまず手入力の支出取引から始める。
- 手入力売上、請求書管理、AI風コメントは後続拡張とする。
- 請求書・レシートAI入力は、ユーザー確認を必須にする。
- グラフは最初はHTML/CSSまたは簡易テーブルで実装する。
- 標準ライブラリ + SQLite のまま進め、Flask等への移行は今回の範囲外とする。
