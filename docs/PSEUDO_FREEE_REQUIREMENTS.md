# 疑似freee会計ダッシュボード 要件定義書

> 📌 **発展方針の正本は [`EVOLUTION_PLAN.md`](EVOLUTION_PLAN.md) です。** 本書は pseudo_freee の要件記録です。全体の進め方・採用スタックは正本を優先してください。

## 1. プロジェクト概要

### プロジェクト名

Inventory × Pseudo freee 連携ポートフォリオ

### 対象アプリ

`pseudo_freee`

### 目的

在庫管理ダッシュボードで登録した仕入・売上データを、疑似freee会計ダッシュボードへ送信し、会計取引として確認できるようにする。

本物のfreee APIへ接続する前段階として、API連携、送信キュー、取引台帳、送信前レビュー、二重送信防止の流れをローカル環境でデモできる状態を目指す。

## 2. 背景

既存の `freee風 会計ダッシュボード` には、売上・経費・利益・未回収請求・KPI分析を表示する単体会計ダッシュボード構想がある。

今回のポートフォリオでは、その構想を活かしつつ、主軸を以下に変更する。

- CSV手入力中心ではなく、在庫管理ダッシュボードからAPIで会計データを受け取る
- 疑似freee側は、受信した仕入・売上データを会計取引として保存する
- 将来的に本物のfreee APIへ置き換えやすい中間構造にする

## 3. 想定ユーザー

- 小規模EC事業者
- 中小企業の経理担当者
- 在庫管理と会計連携をまとめて確認したい担当者
- ポートフォリオ閲覧者、採用担当者

## 4. 全体業務フロー

```text
inventory_dashboard
  1. 商品を選択
  2. 仕入または売上を登録
  3. 在庫移動台帳へ反映
  4. freee_sync_queueへ送信用JSONを保存
  5. ユーザーが送信前レビューを確認
  6. 疑似freeeへ送信

pseudo_freee
  7. 受信APIでJSONを受け取る
  8. 会計取引として保存
  9. 取引一覧・詳細画面で確認
  10. inventory_dashboard側のキューを送信済みにする
```

## 5. 機能要件

### 5.1 会計取引受信API

`inventory_dashboard` から送信された仕入・売上データを受け取る。

エンドポイント:

```text
POST /api/deals
```

受信対象:

- 仕入データ
- 売上データ
- 取引先情報
- 商品明細
- 税区分
- 支払予定日または入金予定日
- 在庫管理側の送信キューID

### 5.2 二重送信防止

同じ `source_type`、`source_id`、`queue_id` の組み合わせが再送された場合、重複登録しない。

想定動作:

- 初回送信: 新規登録
- 同一データ再送: 既存取引を返す、または重複エラーを返す
- 取消・訂正データ: 将来拡張として別取引または訂正履歴で扱う

### 5.3 取引一覧画面

疑似freeeに保存された取引を一覧表示する。

表示項目:

- 取引ID
- 発生日
- 区分
- 取引先
- 請求書番号または注文番号
- 金額
- 税区分
- 支払・入金予定日
- 送信元
- 登録日時

区分:

- 収入
- 支出

### 5.4 取引詳細画面

各取引の詳細を確認できる。

表示項目:

- 取引基本情報
- 明細行
- 取引先情報
- 送信元データ
- 受信JSON
- 在庫管理側のキューID

### 5.5 KPIカード

会計ダッシュボードとして、受信済みデータを集計する。

表示項目:

- 今月売上
- 今月仕入
- 粗利
- 未入金予定額
- 未払予定額
- 受信取引数

### 5.6 月次推移

取引データを月別に集計し、会計状況を確認できるようにする。

表示項目:

- 月別売上
- 月別仕入
- 月別粗利
- 取引件数

### 5.7 送信元確認

在庫管理側から送られたデータであることが分かるように、以下を保持する。

- source_app
- source_type
- source_id
- queue_id
- payload_json

### 5.8 手入力経費

在庫管理側から連携されない支出も、疑似freee側で手入力できるようにする。

入力項目:

- 発生日
- 支払予定日
- 支払先
- 勘定科目
- 税区分
- 金額
- 摘要
- メモ

手入力経費は、在庫連携取引と同じ取引一覧・詳細・KPI・月次推移に反映する。

### 5.9 経費入力候補マスタ

経費入力では、支払先、勘定科目、税区分を候補から選べるようにする。

- 支払先は検索付き候補入力にする
- 勘定科目は検索付き候補入力にする
- 税区分はドロップダウンにする
- 支払先・勘定科目・税区分はマスタ設定から追加できる
- 勘定科目マスタには標準税区分を持たせる
- 勘定科目を選択した場合、標準税区分を税区分欄に反映する
- 経費入力欄では、支払先・勘定科目を既存マスタから検索・選択する
- 候補にない支払先・勘定科目・税区分は、マスタ設定で追加する

### 5.10 証憑画像アップロード・AI入力候補

将来的に、請求書・レシート画像をアップロードし、AI/OCRで経費入力フォームへ候補を反映できるようにする。

方針:

- AIは会計取引を直接登録しない
- まずフォームへ候補を仮入力する
- ユーザーが確認・修正してから登録する
- 元画像、AI抽出結果、ユーザー修正後データを後から確認できるようにする
- スマホではカメラ撮影または画像選択ができるようにする

詳細仕様は `docs/EXPENSE_CAPTURE_FEATURE_SPEC.md` に記載する。

## 6. 非機能要件

### 6.1 ローカル起動

最初はローカルで動作確認できることを優先する。

想定URL:

```text
inventory_dashboard: http://127.0.0.1:8000
pseudo_freee: http://127.0.0.1:8010
```

### 6.2 データベース

初期実装はSQLiteを使用する。

理由:

- ローカルで確認しやすい
- ポートフォリオMVPとして扱いやすい
- DBサーバーなしでデモできる

将来的にはPostgreSQL移行を想定する。

### 6.3 API連携

`inventory_dashboard` 側は、疑似freeeのAPI URLを環境変数で持つ。

```text
PSEUDO_FREEE_API_URL=http://127.0.0.1:8010
```

### 6.4 将来の本物freee連携

疑似freeeで安定した後、本物のfreee APIに置き換えやすいようにする。

将来必要になるもの:

- OAuth2.0
- アクセストークン更新
- 事業所ID
- 取引先IDマッピング
- 勘定科目マッピング
- 税区分マッピング
- APIエラー処理
- レート制限対応

## 8. API仕様案

### POST /api/deals

疑似freeeへ会計取引を登録する。

リクエスト例:

```json
{
  "queue_id": 12,
  "source_type": "purchase",
  "source_id": 4,
  "payload": {
    "api_target": "freee_accounting_deal",
    "issue_date": "2026-06-12",
    "due_date": "2026-07-31",
    "type": "expense",
    "partner_master_id": 1,
    "partner_name": "東京サプライ",
    "freee_partner_id": "",
    "invoice_no": "P-202606-002",
    "details": [
      {
        "sku": "SKU-USB-C-001",
        "description": "USB-Cケーブル 1m",
        "quantity": 25,
        "unit_price": 490,
        "tax_rate": 10,
        "tax_category": "課税仕入 10%",
        "amount": 12250,
        "account_item_name": "仕入高"
      }
    ]
  }
}
```

レスポンス例:

```json
{
  "ok": true,
  "pseudo_freee_deal_id": 101
}
```

### GET /api/deals

登録済み取引一覧を返す。

### GET /api/deals/{id}

取引詳細を返す。

## 12. 完成イメージ

在庫管理ダッシュボードで仕入・売上を入力すると、疑似freee側に会計取引として反映される。

ユーザーは、在庫側で「送信前レビュー」を確認し、疑似freeeへ送信する。疑似freee側では、取引一覧・詳細・受信JSONを確認できる。

これにより、在庫管理から会計連携までの一連の業務フローをポートフォリオとして説明できる。
